# Гайдлайн по проектированию UI-сообщений об ошибках (Error UX Guide)

## 1. Общие принципы
Любое сообщение об ошибке в интерфейсе должно отвечать пользователю на три вопроса:
1. **Что произошло?** (Без технического сленга и обвинений).
2. **Почему это произошло?** (Если это поможет пользователю понять контекст).
3. **Что делать дальше?** (Обязательное руководство к действию или кнопка CTA).

---

## 2. Стандарт структуры JSON-ответа от API
Чтобы техпис или UX-редактор мог подставить человеческий текст, бэкенд должен отдавать понятную структуру. Разработчикам запрещено «выплевывать» сырые логи сервера (например, NullPointerException) на фронтенд.

### Пример правильного маппинга ошибки:

```json
{
  "error": {
    "code": "AUTH_TOKEN_EXPIRED",
    "technical_message": "Jwt token expired at 2026-05-28T12:00:00Z",
    "user_message": "Время сессии истекло. Пожалуйста, войдите в аккаунт заново.",
    "action_type": "REDIRECT_TO_LOGIN"
  }
}
